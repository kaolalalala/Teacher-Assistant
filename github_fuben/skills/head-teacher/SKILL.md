# SKILL.md - Head Teacher（主控技能）

## 技能名称
head-teacher

## 技能描述
通用教师助手的主控技能，负责理解用户需求、识别学段、读取偏好、识别任务类型、调度子技能、验收结果并统一输出。

## 触发条件
作为主入口，所有用户请求首先由本技能处理。

## 偏好与画像读取

### 偏好读取流程

主 agent 在路由任务前，应先做以下判断：

```
步骤1：读取教师偏好
IF profiles/teacher-profile.md 存在且有内容
   → 提取相关偏好信息
ELSE
   → 使用默认规则

步骤2：读取班级画像
IF profiles/classroom-profile.md 存在且有内容
   → 提取相关画像信息
ELSE
   → 使用默认规则

步骤3：判断相关性
   → 判断偏好/画像是否与当前任务相关
   → 仅传递相关部分给子技能
```

### 使用优先级

必须遵循以下优先级：

```
当前用户明确要求 > 教师偏好 > 班级层面画像 > 通用默认规则
```

### 偏好应用方式

主 agent 应在把任务交给各技能时，把相关偏好作为显式上下文传入：

| 技能 | 传入偏好内容 |
|-----|-------------|
| lesson-planner | 教案风格偏好、语言风格、教学偏好 |
| presentation | PPT风格偏好、文字密度、互动偏好 |
| assignment | 题量偏好、作业风格、分层偏好 |
| admin | 沟通语气偏好、文案长度偏好 |
| tutor | 讲解风格偏好、深度偏好、鼓励偏好 |

### 偏好验收规则

主 agent 必须检查输出是否符合已知偏好：

| 偏好 | 验收要求 |
|-----|---------|
| 教师偏好简洁教案 | 输出不应过于冗长 |
| 班级整体基础偏弱 | 作业不应默认过难 |
| 教师偏好温和沟通 | admin 输出应保持温和语气 |
| 教师偏好启发式讲解 | tutor 输出应多提问引导 |

若不匹配，可触发重生成或整理修正。

### 优雅降级

若偏好文件不存在或内容不完整：

1. 不直接失败
2. 使用通用规则输出
3. 保持系统可用性

## 学段识别规则

### 学段分类

| 学段 | 年级范围 | 年龄段 | 认知特点 |
|-----|---------|-------|---------|
| 小学 | 一年级~六年级 | 6-12岁 | 具体形象思维，注意力短，需趣味引导 |
| 初中 | 七年级~九年级 | 12-15岁 | 抽象思维发展，理解力增强，需结构化学习 |
| 高中 | 高一~高三 | 15-18岁 | 抽象思维成熟，逻辑性强，需深度分析 |

### 学段识别方法

```
步骤1：检查用户是否明确说明学段
IF 输入包含 ("小学" OR "一年级"~"六年级" OR "1年级"~"6年级")
   → 学段 = 小学
ELSE IF 输入包含 ("初中" OR "初一"~"初三" OR "七年级"~"九年级" OR "7年级"~"9年级")
   → 学段 = 初中
ELSE IF 输入包含 ("高中" OR "高一"~"高三" OR "十年级"~"十二年级")
   → 学段 = 高中

步骤2：默认策略
ELSE
   → 学段 = 初中（通用性最强）
```

### 学段风格差异

| 维度 | 小学 | 初中 | 高中 |
|-----|------|------|------|
| 整体风格 | 活泼趣味 | 结构清晰 | 紧凑高效 |
| 语言难度 | 简单口语 | 规范准确 | 专业精炼 |
| 内容密度 | 低 | 中 | 高 |
| 互动频率 | 高 | 中 | 适中 |

### 学段传递规则

主 agent 在把任务交给各模块时，**必须**把"学段信息"作为显式上下文传入：

| 技能 | 学段适配要点 |
|-----|-------------|
| lesson-planner | 按学段调整教学目标、课堂设计、语言风格 |
| presentation | 按学段调整文字密度、页面节奏、表达方式 |
| assignment | 按学段调整难度、题型、题量 |
| tutor | 按学段调整讲解深度、语言风格 |
| admin | 按学段调整家校沟通语气 |

### 学段风格验收

主 agent 必须检查结果是否符合学段特征：

| 学段 | 验收要点 |
|-----|---------|
| 小学 | 不能过于抽象、过长、过密；必须活泼有趣 |
| 初中 | 必须结构清晰、重点明确；保持平衡性 |
| 高中 | 不能过于幼态、过浅；必须逻辑严密 |

若风格明显不匹配，应触发重生成或进行整理修正。

## 工作流包识别与编排

### 工作流包识别

当用户请求明显属于"组合型高层任务"时，主 agent 应进入"工作流包模式"：

```
优先级1：明确的工作流包关键词
IF 输入包含 ("备课包" OR "完整备课" OR "一整套课" OR "教学整包")
   → 备课包

ELSE IF 输入包含 ("家校沟通包" OR "一套家长沟通" OR "通知+反馈" OR "家长会内容包")
   → 家校沟通包

ELSE IF 输入包含 ("辅导包" OR "一套辅导内容" OR "讲解+练习" OR "学生辅导整包")
   → 辅导包

ELSE IF 输入包含 ("轻量课堂包" OR "快速备课" OR "简版课堂")
   → 轻量课堂包

优先级2：组合型请求推断
ELSE IF 输入包含 ("备课" OR "教案") AND 包含 ("PPT" OR "课件") AND 包含 ("练习" OR "作业")
   → 备课包

ELSE IF 输入包含 ("通知") AND 包含 ("反馈")
   → 家校沟通包

ELSE IF 输入包含 ("讲解" OR "辅导") AND 包含 ("练习" OR "巩固")
   → 辅导包

优先级3：基础任务（非工作流包）
ELSE
   → 按原有规则识别基础任务
```

### 工作流包编排规则

#### 备课包

**执行顺序**：
```
lesson-planner → presentation → assignment
```

**流程**：
1. 调用 lesson-planner，传入学段、偏好 → 获取教案
2. 提取教案摘要
3. 调用 presentation，传入教案摘要 → 获取PPT
4. 调用 assignment，传入教案摘要 → 获取练习
5. 整合输出：教案 → PPT → 练习

**裁剪规则**：
| 用户要求 | 执行调整 |
|---------|---------|
| 不要PPT | 跳过 presentation |
| 不要练习 | 跳过 assignment |
| 只要教案 | 只执行 lesson-planner |
| 课后作业 | assignment 改为课后作业模式 |

#### 家校沟通包

**执行顺序**：
```
admin(通知) → admin(反馈) → 沟通建议
```

**流程**：
1. 调用 admin，传入类型=通知 → 获取通知草稿
2. 调用 admin，传入类型=反馈 → 获取反馈草稿
3. 生成沟通建议
4. 整合输出：通知 → 反馈 → 建议

#### 辅导包

**执行顺序**：
```
tutor → assignment
```

**流程**：
1. 调用 tutor，传入题目、学段 → 获取讲解内容
2. 提取知识点
3. 调用 assignment，传入知识点 → 获取巩固练习
4. 整合输出：错因分析 → 讲解 → 练习 → 建议

#### 轻量课堂包

**执行顺序**：
```
lesson-planner(简版) → presentation(简版) → assignment(少量)
```

**流程**：
1. 调用 lesson-planner，传入模式=简洁 → 获取简版教案
2. 调用 presentation，传入模式=简洁 → 获取精简PPT
3. 调用 assignment，传入数量=3-4 → 获取少量练习
4. 整合输出

### 工作流包验收规则

主 agent 必须检查：

| 检查项 | 要求 |
|-------|------|
| 完整性 | 所有应有子产物都已生成 |
| 顺序性 | 输出顺序符合定义 |
| 一致性 | 不同模块语气和风格统一 |
| 无冲突 | 不存在明显重复或冲突 |
| 适配性 | 符合对应学段与教师偏好 |

### 风格统一要求

工作流包内所有子模块必须：
- 使用相同的学段设置
- 使用相同的教师偏好
- 保持语言风格一致
- 避免重复内容

## 任务识别规则

### 任务类型判断

根据用户输入中的关键词判断任务类型：

```
优先级1：大复合任务判断
IF 输入包含 ("备课" OR "教案") AND 包含 ("PPT" OR "课件") AND 包含 ("练习" OR "作业" OR "题")
   → 大复合任务（教案+PPT+练习）

优先级2：复合任务判断
ELSE IF 输入包含 ("备课" OR "教案" OR "课堂设计") 
   AND 包含 ("PPT" OR "课件" OR "幻灯片")
   → 复合任务（教案+PPT）

ELSE IF 输入包含 ("备课" OR "教案") AND 包含 ("练习" OR "作业" OR "题")
   → 复合任务（教案+练习）

优先级3：教学类任务
ELSE IF 输入包含 ("备课" OR "教案" OR "课堂设计" OR "教学设计")
   → 教案任务

ELSE IF 输入包含 ("PPT" OR "课件" OR "幻灯片" OR "讲课课件" OR "讲课页")
   → PPT任务

优先级4：作业/练习任务
ELSE IF 输入包含 ("练习" OR "作业" OR "习题" OR "巩固题" OR "分层题" OR "出题" OR "出几道")
   → 作业/练习任务

优先级5：行政类任务
ELSE IF 输入包含 ("通知" OR "评语" OR "活动文案" OR "家校沟通" 
        OR "家长会" OR "班级总结" OR "写给家长" OR "发群")
   → 行政文案任务

优先级6：辅导类任务
ELSE IF 输入包含 ("讲题" OR "讲解" OR "错因" OR "辅导" 
        OR "分析错误" OR "怎么做" OR "解题思路")
   → 学生辅导任务

优先级7：兜底
ELSE
   → 询问用户具体需求
```

### 六类任务对照表

| 任务类型 | 触发关键词 | 调度目标 | 输出特点 |
|---------|-----------|---------|---------|
| 教案 | 备课、教案、课堂设计、教学设计 | lesson-planner | 6部分结构化教案 |
| PPT成品 | PPT、课件、幻灯片、讲课课件 | presentation | 逐页成品内容包 |
| 教案+PPT | 同时包含上述两类 | lesson-planner → presentation | 教案+PPT成品 |
| 行政文案 | 通知、评语、家长会、家校沟通 | admin | 草稿，需确认 |
| 学生辅导 | 讲题、错因、辅导、解题思路 | tutor | 启发式讲解 |
| 作业/练习 | 练习、作业、习题、巩固题、分层题、出题 | assignment | 结构化题目+答案 |

## 信息不足时的兜底规则

### 核心原则

**不要卡住**：当信息不足时，优先产出通用草案，而非反复询问。

### 兜底策略

#### 策略1：自动补全最小上下文

| 任务类型 | 缺失字段 | 默认值 |
|---------|---------|-------|
| 教学类 | 学科 | 语文 |
| 教学类 | 年级 | 初中 |
| 教学类 | 课时 | 40分钟 |
| 作业/练习 | 学科 | 语文 |
| 作业/练习 | 年级 | 初中 |
| 作业/练习 | 题目数量 | 5题 |
| 作业/练习 | 使用场景 | 课堂练习 |
| 行政类 | 对象 | 家长 |
| 行政类 | 语气 | 正式 |
| 辅导类 | 学科 | 数学 |
| 辅导类 | 学生水平 | 中等 |

#### 策略2：产出通用草案

当无法确定具体参数时，生成通用结构草案，并在输出末尾添加提示。

### 不同场景的兜底处理

| 场景 | 用户输入示例 | 处理方式 |
|-----|------------|---------|
| 只有课题 | "做个《春》的PPT" | 自动补全默认值，生成PPT成品内容包 |
| 作业信息模糊 | "出几道题" | 生成通用基础版练习 |
| 行政信息模糊 | "写个通知" | 生成通用通知模板 |
| 辅导信息模糊 | "帮我讲道题" | 询问具体题目 |
| 完全无法推断 | "帮我一下" | 询问具体需求 |

## 调度规则

### 场景A：教案任务

```
1. 提取任务参数（学科、年级、课题、课时）
2. 调用 lesson-planner skill
3. 等待教案生成完成
4. 验收教案字段完整性
5. 整理输出给用户
```

### 场景B：PPT任务

```
1. 提取任务参数
2. 如缺少教学上下文，补全最小信息
3. 调用 presentation skill
4. 等待PPT成品内容包生成完成
5. 验收每页内容完整性
6. 整理输出给用户
```

### 场景C：复合任务（教案+PPT）

```
1. 提取任务参数
2. 调用 lesson-planner skill
3. 等待教案生成完成
4. 验收教案字段完整性
5. 抽取教案摘要
6. 调用 presentation skill，传入教案摘要
7. 等待PPT成品内容包生成完成
8. 验收每页内容完整性
9. 合并输出
```

### 场景D：行政文案任务

```
1. 提取任务参数（对象、场景、语气）
2. 调用 admin skill
3. 等待文案生成完成
4. 验收文案结构完整性
5. 标记为"草稿/待确认"
6. 整理输出给用户
```

### 场景E：学生辅导任务

```
1. 提取任务参数（题目、学科、学生水平）
2. 调用 tutor skill
3. 等待辅导内容生成完成
4. 验收辅导结构完整性
5. 整理输出给用户
```

### 场景F：作业/练习任务

```
1. 提取任务参数（学科、年级、课题、数量、难度、场景）
2. 调用 assignment skill
3. 等待练习生成完成
4. 验收题目结构和数量
5. 整理输出给用户
```

### 场景G：复合任务（教案+练习）

```
1. 提取任务参数
2. 调用 lesson-planner skill
3. 等待教案生成完成
4. 验收教案字段完整性
5. 抽取教案摘要（课题、知识点、教学重点）
6. 调用 assignment skill，传入教案摘要
7. 等待练习生成完成
8. 验收题目与教案一致性
9. 合并输出
```

### 场景H：大复合任务（教案+PPT+练习）

```
1. 提取任务参数
2. 调用 lesson-planner skill
3. 等待教案生成完成
4. 验收教案字段完整性
5. 抽取教案摘要
6. 调用 presentation skill，传入教案摘要
7. 等待PPT成品内容包生成完成
8. 验收PPT内容完整性
9. 调用 assignment skill，传入教案摘要
10. 等待练习生成完成
11. 验收题目与教案一致性
12. 按顺序统一输出
```

**强制要求**：
- 必须串行执行，不允许并行
- 不允许跳过前置内容
- 作业/练习必须基于前置内容生成

## 模式判断与上下文门控

### 模式判断流程

主 agent 在处理每次任务前，必须先执行模式判断：

```
步骤1：读取模式状态
   → 读取 profiles/mode-state.md
   → 获取当前模式状态（ON/OFF）

步骤2：判断模式状态
IF 模式 == ON
   → 进入"教师助手模式"处理流程
ELSE
   → 进入"通用模式"处理流程

步骤3：检查是否为模式切换口令
IF 输入匹配 "开启教师助手" OR "打开教师助手模式" OR "进入教师助手模式"
   → 调用 mode-switch 技能，设置模式为 ON
   → 返回确认信息

ELSE IF 输入匹配 "关闭教师助手" OR "退出教师助手模式" OR "关闭教师助手模式"
   → 调用 mode-switch 技能，设置模式为 OFF
   → 返回确认信息
```

### 模式为 ON 时的默认读取顺序

当模式为 ON 时，主 agent 在处理任务前应按以下顺序读取：

| 顺序 | 文件路径 | 用途 | 读取时机 |
|-----|---------|------|---------|
| 1 | `profiles/mode-state.md` | 确认当前模式 | 每次任务开始时 |
| 2 | `MEMORY.md` | 高频核心偏好 | 每次任务开始时 |
| 3 | `profiles/teacher-profile.md` | 教师详细偏好 | 任务开始时按需 |
| 4 | `profiles/classroom-profile.md` | 班级画像 | 任务开始时按需 |
| 5 | `profiles/defaults.md` | 系统默认配置 | 偏好缺失时 |

然后再根据任务类型决定是否按需参考 `memory/` 目录：

| 目录 | 读取时机 |
|-----|---------|
| `memory/lessons/` | 处理该课题任务时 |
| `memory/classes/` | 处理该班级任务时 |
| `memory/packs/` | 优化工作流包时 |
| `memory/daily/` | 查找特定日期记录时 |
| `memory/weekly/` | 回顾周期趋势时 |

**重要**：`memory/` 目录只按需读取，不每轮全量读取。

### 模式为 OFF 时的规则

当模式为 OFF 时：

- 不默认读取上述项目记忆文件
- 直接按通用任务逻辑处理
- 只有用户明确要求时才读取特定文件

### 优先级规则

无论模式是否开启，都必须遵循以下优先级：

```
当前用户明确要求 > 当前模式状态 > 教师偏好/班级画像 > 通用默认规则
```

### 模式开关不覆盖当前指令

即使教师助手模式为 ON，若用户当前明确说：

- "不要参考历史偏好"
- "临时按另一种风格生成"
- "这次不要用教师助手模式"

则必须优先服从当前要求。

### 模式切换确认

当用户发送模式切换口令时，主 agent 应：

**开启教师助手**：
```
1. 调用 mode-switch 技能
2. 更新 profiles/mode-state.md 状态为 ON
3. 返回确认：
   - 已进入教师助手模式
   - 后续将默认参考教师助手项目记忆与规则
```

**关闭教师助手**：
```
1. 调用 mode-switch 技能
2. 更新 profiles/mode-state.md 状态为 OFF
3. 返回确认：
   - 已退出教师助手模式
   - 后续将不再默认读取项目记忆文件
```

```

主 agent 处理：

6. 验收并输出

主 agent 处理：
1. 识别为模式切换口令
2. 调用 mode-switch 技能
3. 更新 profiles/mode-state.md 状态为 OFF
4. 返回确认信息
```

## 最终输出治理

### 治理职责

主 agent 在汇总任何结果时，必须执行最终输出治理：

| 检查项 | 要求 | 处理方式 |
|-------|------|---------|
| 标题统一 | 各模块标题风格一致 | 统一为 `{类型}：{主题}` 格式 |
| 层级一致 | 小节层级不混乱 | 确保H1/H2/H3层级正确 |
| 内容去重 | 删除各模块间重复说明 | 保留首次出现，删除后续重复 |
| 语气连贯 | 无明显语气割裂 | 统一专业、清晰的语气 |
| 长度适中 | 无过长、过短、风格失衡 | 调整内容密度 |

### 整合流程

```
步骤1：收集所有子模块输出
步骤2：检查标题格式
   → 统一为 "# {类型}：{主题}"
步骤3：检查小节层级
   → 确保 H2 为模块名，H3 为子部分
步骤4：去重处理
   → 删除重复的说明文字
   → 合并相似的提示语
步骤5：添加导读（如适用）
   → 输出较长或来自多模块时添加
步骤6：统一结尾
   → 整包只需一个结尾
步骤7：最终输出
```

### 导读生成规则

**触发条件**：
- 输出内容超过 100 行
- 来自多个模块的整包输出
- 工作流包结果

**格式**：
```markdown
> **导读**：本{类型}包含{内容概要}，共{数量}部分。
```

**示例**：
```markdown
> **导读**：本备课包包含教案、PPT课件、课堂练习三部分，已按初中风格生成。
```

### 结尾统一规则

**单模块输出**：
```markdown
---
*本内容由教师助手生成，可根据实际需要调整。*
```

**多模块整包输出**：
```markdown
---
*本备课包由教师助手生成，包含教案、PPT、练习三部分，可根据实际需要调整使用。*
```

**禁止**：
- ❌ 每个子模块都加结尾
- ❌ 过长的使用说明
- ❌ 重复的"如需我继续..."

模块之间使用简短过渡语：

```markdown
---

## 二、PPT课件

以下是配套PPT课件内容：
```

如果结果明显像多个模块硬拼接：
- 应重新整理
- 必要时要求局部重生成
- 最终必须达到"整体一致、阅读顺畅"的标准

## 评测模式支持

### 评测模式职责

当进入评测场景时，主 agent 应能够：

1. 按固定测试样例执行
2. 判断结果是否满足核心预期
3. 记录是否通过/部分通过/不通过
4. 指出明显退化点

### 评测模式触发

评测模式可通过以下方式触发：

| 触发方式 | 说明 |
|---------|------|
| 用户明确要求 | "运行回归测试"、"执行评测" |
| 规则更新后 | 更新 AGENTS.md 或 SKILL.md 后建议执行 |
| 阶段验收时 | 每个阶段完成后执行回归检查 |

### 核心判断维度

评测时至少检查以下内容：

| 检查维度 | 说明 | 判定标准 |
|---------|------|---------|
| 路由正确性 | 是否路由到正确模块 | 任务类型与调度目标匹配 |
| 输出结构 | 是否输出了应有结构 | 必需字段完整 |
| 降级触发 | 是否触发了不该发生的降级 | 无异常降级 |
| 模式开关行为 | 模式 ON/OFF 是否正确 | 行为符合模式状态 |
| 工作流包顺序 | 顺序是否正确 | 按定义顺序执行 |
| 已有能力完整性 | 是否破坏之前已通过的能力 | 历史用例仍通过 |

### 评测结果分类

#### Pass（通过）

- 核心行为符合预期
- 输出结构完整
- 无明显退化

#### Partial Pass（部分通过）

- 能完成任务
- 但结构、风格或顺序有轻微偏差
- 不影响主能力可用性

#### Fail（不通过）

- 路由错误
- 结构缺失严重
- 模式开关失效
- 工作流顺序错误
- 核心功能不可用

### 评测执行流程

```
步骤1：读取测试用例
   → 从 eval/cases/ 读取固定测试用例
   → 按用例顺序执行

步骤2：执行测试用例
   → 按用例输入指令执行任务
   → 记录实际输出

步骤3：对照 baseline 检查
   → 读取 eval/baselines/ 中的预期行为
   → 对比实际行为与预期行为

步骤4：判定结果
   → Pass / Partial Pass / Fail
   → 记录偏差点

步骤5：记录评测结果
   → 写入 eval/results/
   → 包含问题摘要和建议修复项
```

### 评测注意事项

1. **不要求逐字一致**：评测重点是行为正确性和结构完整性
2. **允许轻微措辞变化**：只要行为正确、结构合理，即可视为通过或部分通过
3. **关注退化**：重点检查是否破坏已有能力
4. **结果可追溯**：每轮评测都留下结果文件

### 评测用例覆盖

评测应覆盖以下类型：

| 用例类型 | 覆盖场景 |
|---------|---------|
| 单独教案 | 教案生成能力 |
| 单独 PPT | PPT 生成能力 |
| 单独作业 | 作业/练习生成能力 |

## 模型选择规则

### 模型选择职责

主 agent 在确定任务类型后，应进一步判断：

1. 该任务的默认模型是什么
2. 是否属于复杂任务
3. 是否属于低成本可降级任务
4. 是否属于工作流包中的不同阶段任务

### 模型选择流程

```
步骤1：确定任务类型
   → 识别任务类型（教案/PPT/作业/行政/辅导/工作流包）

步骤2：读取模型路由配置
   → 读取 profiles/model-routing.md
   → 获取任务类型对应的默认模型层级

步骤3：检查用户是否有明确要求
   IF 用户明确要求使用特定模型
      → 优先使用用户指定的模型
   ELSE
      → 使用默认模型层级

步骤4：检查模型可用性
   IF 目标模型可用
      → 使用目标模型
   ELSE
      → 尝试备选模型
      IF 备选模型可用
         → 使用备选模型
      ELSE
         → 使用当前可用主模型
         → 记录降级事件
      END
   END

步骤5：执行任务
   → 使用选定的模型执行任务
```

### 默认模型映射

| 任务类型 | 默认模型层级 | 备选模型层级 | 优先级 |
|---------|-------------|-------------|-------|
| 教案生成 | Tier-1 | Tier-2 | 质量优先 |
| PPT/课件 | Tier-1 | Tier-2 | 质量优先 |
| 学生辅导 | Tier-1 | Tier-2 | 质量优先 |
| 作业/练习 | Tier-2 | Tier-3 | 平衡 |
| 行政文案 | Tier-3 | Tier-2 | 成本优先 |
| 观察记录 | Tier-3 | - | 成本优先 |
| 评测结果整理 | Tier-3 | - | 成本优先 |

### 工作流包模型分层

工作流包的不同阶段可使用不同模型：

| 阶段 | 模型层级 | 说明 |
|-----|---------|------|
| 总控规划 | Tier-1 | 需要规划能力 |
| 教案生成 | Tier-1 | 核心产出 |
| PPT生成 | Tier-1 | 核心产出 |
| 练习生成 | Tier-2 | 可适度控制成本 |
| 输出整形 | Tier-2 | 格式化任务 |

### 工作流包模型选择示例

**备课包执行流程**：

```
1. 总控规划（Tier-1）
   → 分析用户需求
   → 规划执行顺序

2. 教案生成（Tier-1）
   → 调用 lesson-planner
   → 使用强模型生成教案

3. PPT生成（Tier-1）
   → 调用 presentation
   → 使用强模型生成 PPT

4. 练习生成（Tier-2）
   → 调用 assignment
   → 使用中等模型生成练习

5. 输出整形（Tier-2）
   → 整合所有输出
   → 使用中等模型进行格式化
```

### 用户明确指定模型

当用户明确要求使用特定模型或风格时：

| 用户要求 | 处理方式 |
|---------|---------|
| "用最强模型给我写一份教案" | 强制使用 Tier-1 |
| "快速生成一份通知" | 可使用 Tier-3 |
| "这次要高质量输出" | 使用 Tier-1 |
| "简单处理就行" | 可使用 Tier-3 |

### 降级处理

当目标模型不可用时：

```
1. 尝试备选模型
2. 若备选也不可用，使用当前可用主模型
3. 记录降级事件到 logs/errors/
4. 不因模型路由失败而中断主流程
```

### 降级记录格式

```markdown
| 字段 | 内容 |
|-----|------|
| 时间 | YYYY-MM-DD HH:mm:ss |
| 任务类型 | [任务类型] |
| 原始目标模型 | Tier-X |
| 降级原因 | [原因] |
| 最终使用模型 | Tier-Y |
```

### 模型路由优先级

```
用户明确指定模型 > 任务类型默认映射 > 系统负载状态调整 > 备选模型降级 > 兜底主模型
```

| 配置 | `profiles/model-routing.md` | 模型路由配置 |

## 统一主入口路由器

### 主入口路由器职责

head-teacher 是系统的**唯一主入口路由器**，负责：

1. 统一判断请求属于哪一类
2. 处理 skill 边界重叠
3. 按冲突矩阵裁决
4. 按统一兜底策略降级
5. 约束后处理类不抢首轮入口

### 固定判断顺序

主 agent 必须按以下固定顺序判断：

```
步骤1：是否为控制类请求
IF 匹配 mode-switch 或 evaluation 触发词
   → 路由到对应控制类 skill
   → 结束

步骤2：是否为组合类请求
IF 匹配 workflow-pack 触发词 OR 包含多模块组合
   → 路由到 workflow-pack
   → 结束

步骤3：是否为单技能请求
IF 匹配某个单技能触发词
   → 检查是否存在 skill 冲突
   IF 存在冲突
      → 按冲突矩阵裁决
   ELSE
      → 路由到对应单技能
   END
   → 结束

步骤4：是否需要兜底策略
IF 无法明确判断
   → 按关键词兜底规则路由
   → 结束

步骤5：主任务完成后，是否需要后处理 skill
IF 需要整形
   → 调用 quality-governor
IF 需要记录日志
   → 调用 observability
```

### 固定优先级

| 优先级 | 类型 | 说明 |
|-------|------|------|
| 1 | 控制类 | 优先拦截 |
| 2 | 组合类 | 次优先 |
| 3 | 单技能类 | 其后 |
| 4 | 后处理类 | 只后置参与 |

### 冲突裁决规则

#### lesson-planner vs workflow-pack

| 用户请求 | 路由目标 | 判断依据 |
|---------|---------|---------|
| "给我备一节课" | lesson-planner | 单独备课 |
| "帮我做个教案" | lesson-planner | 单独教案 |
| "给我准备一整套课" | workflow-pack | 组合型请求 |
| "备课包" | workflow-pack | 组合型请求 |
| "教案 + PPT" | workflow-pack | 多模块组合 |

#### presentation vs workflow-pack

| 用户请求 | 路由目标 | 判断依据 |
|---------|---------|---------|
| "给我做个PPT" | presentation | 纯PPT |
| "帮我做个课件" | presentation | 纯课件 |
| "备课 + PPT" | workflow-pack | 组合请求 |

#### assignment vs tutor

| 用户请求 | 路由目标 | 判断依据 |
|---------|---------|---------|
| "出5道练习题" | assignment | 纯出题 |
| "布置一份作业" | assignment | 纯作业 |
| "帮我讲这道题" | tutor | 讲解请求 |
| "分析一下错因" | tutor | 辅导请求 |
| "讲解 + 出类似题" | workflow-pack | 组合请求（辅导包） |
| "不要直接给答案" | tutor | 启发式辅导特征 |

#### admin vs workflow-pack

| 用户请求 | 路由目标 | 判断依据 |
|---------|---------|---------|
| "写个通知" | admin | 单独通知 |
| "帮我写评语" | admin | 单独评语 |
| "通知 + 反馈一起做" | workflow-pack | 组合请求 |
| "家校沟通包" | workflow-pack | 组合请求 |

#### 控制类 vs 业务类

- 一律控制类优先

### 兜底规则

当无法明确判断时，按关键词兜底：

| 关键词 | 兜底路由 |
|-------|---------|
| 备课、教案 | lesson-planner |
| PPT、课件、幻灯片 | presentation |
| 作业、练习、题、习题 | assignment |
| 讲解、错因、不要直接给答案 | tutor |
| 通知、评语、家长、家校沟通 | admin |
| 做一套教学内容 | workflow-pack（通用降级版） |

### 后处理类约束

后处理类 skill（quality-governor、observability）**不得**抢占首轮主路由：

- ❌ 作为首轮路由目标
- ❌ 抢占业务类 skill
- ❌ 在主任务未完成时介入

正确流程：
```
1. 首轮路由 → 业务类 skill
2. 执行主任务
3. 主任务完成后 → 检查是否需要后处理
4. 如需整形 → quality-governor
5. 如需记录 → observability
```

## Skill 路由后的模型路由

### 模型路由职责

head-teacher 在完成 skill 路由后，必须进一步进行模型路由判断：

1. 先确定使用哪个 skill
2. 再根据 skill 类型选择默认模型档位
3. 再根据任务复杂度判断是否升级或降级
4. 最后将模型选择结果传给对应 skill

### 固定判断顺序

```
步骤1：完成 Skill 路由
   → 按路由规则确定目标 skill
   → 获取 skill 类型

步骤2：查询默认模型档位
   → 读取 models/model-map.md
   → 获取该 skill 的默认模型档位

步骤3：检查历史模型使用
   IF 同类任务有历史记录
      → 优先使用历史档位（稳定性优先）
   ELSE
      → 使用默认档位

步骤4：评估任务复杂度
   IF 任务复杂度高
      → 检查是否满足升级条件
      IF 满足升级条件
         → 升级到更高档位
   ELSE IF 任务复杂度低
      → 检查是否允许降级
      IF 允许降级且满足降级条件
         → 降级到更低档位

步骤5：检查稳定性约束
   IF 同类任务档位波动频繁
      → 锁定默认档位
   IF 升降级来回切换
      → 冻结档位选择

步骤6：检查成本预算
   IF 预算紧张
      → 评估降级可能
   IF 预算充足
      → 执行选择

步骤7：输出模型选择结果
   → 将模型档位传给目标 skill
   → 记录选择原因
```

### 默认模型原则

| Skill | 默认档位 | 说明 |
|-------|----------|------|
| workflow-pack | strong | 组合编排需强模型保障一致性 |
| lesson-planner | standard | 详细教案可升强模型 |
| presentation | standard | 长篇PPT可升强模型 |
| assignment | standard | 复杂分层题可升强模型 |
| tutor | standard | 复杂讲解可升强模型，不降级 |
| admin | light | 正式通知可升标准模型 |
| quality-governor | light | 仅整形，不需强模型 |
| observability | light | 只做摘要记录 |
| evaluation | standard | 复杂评估可升强模型 |
| mode-switch | light | 只处理模式口令 |
| router-governor | standard | 路由冲突复杂可升强模型 |
| model-governor | standard | 模型路由判断需稳定 |

### 升级规则

满足以下条件时升级到强模型：

| 触发场景 | 判断标准 | 适用 Skill |
|----------|----------|------------|
| 组合型工作流 | 多 Skill 编排、整包输出 | workflow-pack |
| 长文本整合 | 输出预计 >2000字 | lesson-planner, presentation |
| 结构复杂 | 多层级、多模块结构 | lesson-planner, presentation, assignment |
| 强一致性要求 | 风格/格式需高度统一 | workflow-pack, presentation |
| 复杂推理 | 多步骤逻辑链 | tutor, evaluation, router-governor |
| 启发式讲解 | 需要深度引导设计 | tutor |
| 复杂评估 | 多维度综合判断 | evaluation |
| 路由冲突严重 | 多 Skill 边界模糊 | router-governor, model-governor |

### 降级规则

满足以下条件时可降级到轻量模型：

| 触发场景 | 判断标准 | 适用 Skill |
|----------|----------|------------|
| 任务简单 | 复杂度 < 中 | lesson-planner, presentation, assignment |
| 输出短 | 预计输出 <500字 | lesson-planner, presentation |
| 格式整理 | 仅需结构调整 | quality-governor |
| 简单改写 | 文本润色、格式化 | admin |
| 状态切换 | 模式口令处理 | mode-switch |
| 日志记录 | 摘要、统计记录 | observability |
| 简单统计 | 数值汇总 | evaluation |

### 禁止降级场景

以下场景**禁止降级**：

1. **tutor（答疑辅导）**：辅导质量优先，不允许降级到 light
2. **model-governor（模型治理）**：判断稳定性优先，不允许降级
3. **已为最低档位**：light 档位不允许继续降级
4. **用户明确要求高质量**：尊重用户期望
5. **历史同类任务使用高档模型**：保持一致性

### 同类任务模型稳定性

主 agent 必须保障同类任务的模型稳定性：

| 稳定性规则 | 说明 |
|-----------|------|
| 同类任务使用相同档位 | 避免同类任务频繁换模 |
| 同一会话保持一致 | 同一会话内模型档位稳定 |
| 同用户跨会话风格统一 | 保持风格一致性 |
| 模型切换需记录原因 | 每次切换必须有明确理由 |

### 频繁换模检测

当检测到以下情况时，触发稳定性警告：

| 检测项 | 阈值 | 处理方式 |
|--------|------|----------|
| 同类任务档位波动 | 3次/会话 | 锁定默认档位 |
| 同一Skill换模频率 | >50%/会话 | 强制使用默认档位 |
| 升降级来回切换 | 2次/任务流 | 冻结档位选择 |

### 模型路由示例

**示例1：教案任务（标准复杂度）**

```
用户：帮我备一节《春》的课

处理流程：
1. Skill 路由 → lesson-planner
2. 查询默认档位 → standard
3. 检查历史 → 无历史
4. 评估复杂度 → 中等
5. 稳定性检查 → 通过
6. 最终档位 → standard
```

**示例2：复杂教案任务（升级）**

```
用户：帮我设计一份完整的跨学科教学设计，包含多课时整合

处理流程：
1. Skill 路由 → lesson-planner
2. 查询默认档位 → standard
3. 检查历史 → 无历史
4. 评估复杂度 → 高（跨学科、多课时）
5. 检查升级条件 → 满足"长文本整合"
6. 最终档位 → strong（升级）
```

**示例3：简单通知任务（降级）**

```
用户：写个简短的通知，提醒明天穿校服

处理流程：
1. Skill 路由 → admin
2. 查询默认档位 → light
3. 检查历史 → 无历史
4. 评估复杂度 → 低（简短通知）
5. 最终档位 → light（保持默认）
```

**示例4：辅导任务（禁止降级）**

```
用户：简单讲一下这道题

处理流程：
1. Skill 路由 → tutor
2. 查询默认档位 → standard
3. 检查历史 → 无历史
4. 评估复杂度 → 低
5. 检查降级条件 → 禁止降级（tutor 不允许降级到 light）
6. 最终档位 → standard（保持默认）
```
