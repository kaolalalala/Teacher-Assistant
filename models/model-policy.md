# 模型策略文件（Model Policy）

> 本文件定义模型升级/降级策略、同类任务模型稳定性规则。
> 多模型路由稳定化

---

## 一、模型选择核心原则

### 1.1 四维平衡原则

模型选择必须兼顾以下四个维度：

| 维度 | 权重 | 说明 |
|------|------|------|
| 任务复杂度 | 35% | 复杂任务需要强模型保障质量 |
| 输出稳定性 | 30% | 同类任务应保持模型一致性 |
| 成本控制 | 20% | 避免无意义占用高档模型 |
| 风格一致性 | 15% | 同一用户/会话保持风格统一 |

### 1.2 默认优先原则

- **同类任务优先使用默认档位**
- 不允许同类任务无规则频繁换模
- 升级/降级必须有明确触发条件
- 临时实验性调整需记录原因

## 二、模型升级策略

### 2.1 升级触发条件

#### 升级到强模型（standard → strong）

| 触发场景 | 判断标准 | 适用 Skill |
|----------|----------|------------|
| 组合型工作流 | 多 Skill 编排、整包输出 | workflow-pack |
| 长文本整合 | 输出预计 >2000字 | lesson-planner, presentation |
| 结构复杂 | 多层级、多模块结构 | lesson-planner, presentation, assignment |
| 强一致性要求 | 风格/格式需高度统一 | workflow-pack, presentation |
| 复杂推理 | 多步骤逻辑链 | tutor, evaluation, router-governor |
| 启发式讲解 | 需要深度引导设计 | tutor |
| 复杂评估 | 多维度综合判断 | evaluation |
| 路由冲突严重 | 多 Skill 边界模糊 | router-governor, model-governor |

#### 升级判断流程

```
1. 检查任务复杂度
   ├── 复杂度 >= 高 → 进入升级评估
   └── 复杂度 < 高 → 保持默认

2. 评估升级收益
   ├── 质量提升明显 → 允许升级
   └── 质量提升有限 → 保持默认

3. 检查成本预算
   ├── 预算充足 → 执行升级
   └── 预算紧张 → 记录并保持默认
```

### 2.2 升级决策矩阵

| 当前档位 | 任务复杂度 | 输出长度 | 一致性要求 | 决策 |
|----------|------------|----------|------------|------|
| standard | 高 | 长 | 高 | 升级到 strong |
| standard | 高 | 长 | 中 | 升级到 strong |
| standard | 高 | 短 | 高 | 保持 standard |
| standard | 中 | 长 | 高 | 保持 standard |
| standard | 中 | 中 | 中 | 保持 standard |
| light | 高 | 长 | 高 | 升级到 standard |
| light | 中 | 中 | 中 | 保持 light |

## 三、模型降级策略

### 3.1 降级触发条件

#### 降级到轻量模型（standard → light）

| 触发场景 | 判断标准 | 适用 Skill |
|----------|----------|------------|
| 任务简单 | 复杂度 < 中 | lesson-planner, presentation, assignment |
| 输出短 | 预计输出 <500字 | lesson-planner, presentation |
| 格式整理 | 仅需结构调整 | quality-governor |
| 简单改写 | 文本润色、格式化 | admin |
| 状态切换 | 模式口令处理 | mode-switch |
| 日志记录 | 摘要、统计记录 | observability |
| 简单统计 | 数值汇总 | evaluation |

#### 降级判断流程

```
1. 检查任务复杂度
   ├── 复杂度 <= 低 → 进入降级评估
   └── 复杂度 > 低 → 保持默认

2. 评估降级风险
   ├── 质量影响可接受 → 允许降级
   └── 质量影响明显 → 保持默认

3. 检查用户期望
   ├── 用户无特殊要求 → 执行降级
   └── 用户期望高质量 → 保持默认
```

### 3.2 降级决策矩阵

| 当前档位 | 任务复杂度 | 输出长度 | 质量要求 | 决策 |
|----------|------------|----------|----------|------|
| standard | 低 | 短 | 低 | 降级到 light |
| standard | 低 | 短 | 中 | 保持 standard |
| standard | 低 | 长 | 低 | 保持 standard |
| standard | 中 | 短 | 低 | 保持 standard |
| strong | 低 | 短 | 低 | 降级到 standard |
| strong | 中 | 中 | 中 | 保持 strong |

### 3.3 禁止降级场景

以下场景**禁止降级**：

1. **tutor（答疑辅导）**：辅导质量优先，不允许降级到 light
2. **model-governor（模型治理）**：判断稳定性优先，不允许降级
3. **已为最低档位**：light 档位不允许继续降级
4. **用户明确要求高质量**：尊重用户期望
5. **历史同类任务使用高档模型**：保持一致性

## 四、同类任务模型稳定性规则

### 4.1 稳定性原则

```
原则1：同类任务使用相同模型档位
原则2：同一会话内保持模型一致性
原则3：同一用户跨会话保持风格统一
原则4：模型切换需有明确理由并记录
```

### 4.2 稳定性检查流程

```
1. 检查任务类型
   └── 确定 Skill 分类

2. 查询历史模型使用
   ├── 同类任务有历史记录 → 优先使用历史档位
   └── 无历史记录 → 使用默认档位

3. 评估是否需要调整
   ├── 满足升级/降级条件 → 调整并记录原因
   └── 不满足条件 → 保持历史/默认档位

4. 记录本次选择
   └── 写入模型使用日志
```

### 4.3 频繁换模检测

当检测到以下情况时，触发稳定性警告：

| 检测项 | 阈值 | 处理方式 |
|--------|------|----------|
| 同类任务档位波动 | 3次/会话 | 锁定默认档位 |
| 同一Skill换模频率 | >50%/会话 | 强制使用默认档位 |
| 升降级来回切换 | 2次/任务流 | 冻结档位选择 |

## 五、成本控制策略

### 5.1 成本预算分配

| 档位 | 单次成本系数 | 建议使用比例 |
|------|--------------|--------------|
| strong | 3.0 | ≤20% |
| standard | 1.5 | 50-60% |
| light | 1.0 | ≥20% |

### 5.2 成本控制规则

1. **强模型配额限制**：每会话强模型使用不超过 20%
2. **批量任务降级**：批量生成任务优先使用轻量模型
3. **成本预警**：单会话成本超阈值时自动降级
4. **用户配额**：支持按用户设置成本上限

## 六、模型选择决策树

```
开始模型选择
│
├── 1. 确定 Skill 类型
│   └── 查询 model-map.md 获取默认档位
│
├── 2. 检查历史使用
│   ├── 有同类任务历史 → 优先使用历史档位
│   └── 无历史 → 使用默认档位
│
├── 3. 评估任务复杂度
│   ├── 复杂度高 → 检查升级条件
│   ├── 复杂度中 → 保持当前档位
│   └── 复杂度低 → 检查降级条件
│
├── 4. 检查稳定性约束
│   ├── 同类任务档位波动 → 锁定默认档位
│   └── 档位稳定 → 继续判断
│
├── 5. 检查成本预算
│   ├── 预算充足 → 执行选择
│   └── 预算紧张 → 评估降级可能
│
└── 6. 输出模型选择结果
    └── 记录选择原因
```

## 七、策略执行优先级

| 优先级 | 规则类型 | 说明 |
|--------|----------|------|
| P0 | 禁止规则 | 不推荐档位绝对禁止 |
| P1 | 稳定性规则 | 同类任务保持一致 |
| P2 | 升级规则 | 满足条件才升级 |
| P3 | 降级规则 | 评估影响后降级 |
| P4 | 成本规则 | 预算约束下调整 |
